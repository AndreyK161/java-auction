## Онлайн Аукцион

Задача: Реализовать приложение онлайн аукцион торгов.

[Перечень реализации приложения](online-auction-tasks.md)

### Функциональные требования

#### Лоты:

- В торгах участвуют лоты, которые продаются продавцами. Характеристиками лотов является: название, описание, фотографии (jpg, png), начальная цена, шаг цены, продавец (пользователь)
- Приложение должно иметь возможность зарегистрировать новый лот
- Все зарегистрированные лоты должны храниться в истории приложения

#### Проведение торгов:

- В торгах участвует утвержденный администратором список лотов
- Торги содержат: перечень лотов, перечень участников, дату проведения, статус (опубликованы, проводятся, завершены)
- Лот может участвовать только в одних опубликованных торгах
- Торги публикуются заранее администратором торгов с ориентировочной датой старта
- Допускается публикация множества будущих торгов, но проведение происходит только одних торгов
- Характеристики лота можно изменять только до начала торгов
- Для покупки лота используются ставки с определенным шагом цены
- При объявлении торгов по лоту со стартовой ценой, ожидается прием ставок
- Процесс продажи лота: 1 - начало торгов, 2 - начало торгов по лоту, одновременно могут быть выставлены на продажу
  несколько лотов
- Ставки должны быть выше текущей цены лота и кратные шагу цены указанной в информации по лоту
- По истечении одной минуты времени с момента последней ставки торги по лоту завершаются
- Если торги по лоту завершились и лот не был продан (не было ставок), то лот может быть выставлен на продажу в других торгах, при этому
  допускается изменение его цены
- При завершении торгов по лоту, товар зачисляется участнику с максимальной ставкой, а продавцу зачисляются на ЛС сумма проданного лота
- У победителя торгов по лоту списывается со счета сумма размером с финальной стоимостью лота

#### Участники:

- Регистрируются в системе с указанием почты, на который в последствии отправляется пароль для входа
- Данные участника: почта (уникальная), фио
- Идентифицируются по логину (почте)
- Имеют личный счет. Это нужно для регистрации участников и лотов
- Личный счет участника должен содержать историю движений средств (транзакций)
- Лоты участвуют в торгах в порядке очереди. Не все зарегистрированные лоты должны участвовать в торгах
- Участники могут зарегистрироваться на опубликованные и активные торги
- При регистрации на торги участники вносят депозит на личные счета для расчетов по выигранным лотам
- Участники могут также быть зарегистрированы на уже идущие торги
- При регистрации участника по торгам ему назначается уникальный номер (+хеш), с которым можно делать ставки 
- После проведения торгов должен быть доступен отчет, где указано: время торгов, сумма торгов, победители с их детализацией лотов и финальных цен
- Через час после окончания торгов участникам отправляются отчеты
- Пользователь имеет лс, который можно пополнять и снимать средства (через банковский сервис)
- У участника хранится история участия в торгах и приобретенных лотах

#### Пополнение личного счета:

- Через администратора с указанием суммы
- Через платежную (ПС) систему с указанием суммы пополнения, на которую формируется отдельный заказ (ПС реализовать
  самостоятельно)

#### Онлайн табло
- Для активных торгов должно быть доступно табло, которое отображает: лоты которые в данный момент торгуются с информацией о: данные лота, начало торгов по лоту, количество ставок, текущая принятая ставка, начальная цена.
- Так же должна быть доступна история уже проданных лотов с информацией о лоте, финальной ставке и победителем.
- Табло должно справляться с высокими нагрузками 1M rps

#### Роли приложения:

1. Администратор торгов (запускает торги, пауза, завершение, определяет перечень лотов для торгов, может пополнить ЛС
   участника)
2. Участник торгов. От имени участника подается лот, средства при продаже поступают на ЛС. Участник так же может зарегистрироваться на торги для совершения ставок на лоты. 

#### Нефункциональные требования (*возможности фреймворка):

- реализовать логирование в приложении
- поработать с моделью событий
- приложение должны выдерживать высокие нагрузки (1000 запросов/ставок в секунду)

### Архитектурные требования
1. История изменений должна сохраняться через систему событий
2. Прием ставок является высококонкурентным процессом, поэтому обрабатывать ставки необходимо последовательно
3. Сбор логов запросов ответов должен быть настроен для отдельного файла.
4. Остановка торгов должна быть доступна так же через консоль
